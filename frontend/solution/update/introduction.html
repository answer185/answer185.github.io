# 底层升级方案介绍

# 背景

# 概述

红圈项目始于2016年。立项时，前端行业的主流框架是react, 当时生态圈还不成熟。我们选择了蚂蚁金服的相关技术栈。具体信息如下：

名称 说明 使用版本 当前版本
atool-build 基于webpack 1.x的封装 0.11.1 停止维护
dora 其他webpack的开发服务器封装 0.4.5 停止维护
antd 2.x版本 2.13.11 4.X
dva 数据流方案 1.2.1 2.6.0
react react生态圈 15.6.2 16.13.1

随着前端技术的发展，蚂蚁金服基于他们的最佳实践，做了很大的调整。主要有两个大版本的转换：

  1. atool-build+dora 合并成roadhog的项目，对dva做了升级，更好的和roadhog结合， 扩展了dva的能力。
  2. 为增强路由层和扩展能力，他们创建了新的项目UMI, 它是以roadhog+路由为基础，插件体系为主的项目， 并把dva作为插件结合进来。

# 严重问题

  1. 在2017年底，我们发现node版本最多只能支持到8.9.1, 且该问题是atool-build, dora之类的底层技术引起。 原因主要是因为node版本升级时，不做向下兼容，放弃了某些API。

  2. 随着react的发展，从版本16开始也不向下兼容，导致react无法升级，这也直接也影响到了antd等相关包的升级， 如antd，从3.X开始，已经是基于16开发。 项目的技术发展碰到天花板。

  3. 截至2020年6月5号，项目的模块已经达到5748个。项目启动慢：

  • 第一次启动时间是15分钟左右，
  • 第二次启动，虽然有缓存的优化，还是需要2分钟。
  • 在开发时，修改代码后，重启服务会直接卡死，导致无法快捷开发

极大影响开发效率

当前的解决方案：

  1. 引入多进程打包
  2. 只加载部分路由
  3. 在启动时，去掉自动检测和安装依赖包的功能

但是治标不治本。最新版本的webpack拥有解决该问题的方案

# 升级历史

  1. 2017年底，曾咏杰和刘俊尝试升级(未上报)，但是因为业务层开发占用了太多时间，放弃。

  2. 2018年10月左右，李鹏飞（时任杭州前端团队Leader）再次尝试，但是没做成。不过引入了webpack的多进程打包， 优化了启动速度和打包能力。

  3. 2019年4月，北京前端团队内部启动，代号UMI迁移。但是为保障红圈快消项目开发，和阙耀楚（时任部门技术VP）商量后，未立项。

  4. 2019年11月左右，因为产品发展需要桌面端，且做该升级需要熟悉umi，前端定了更大的方案，尝试Electron+umi的全新技术架构。 在和阙耀楚（时任部门技术VP）讨论后: 我们觉目前的前端团队能力还欠缺，需要先做技术调研和准备，由咏杰负责，前端团队全体参与， 先尝试做一个桌面端的demo。后来因为业务层开发，支援平台研发部门的Welink开发及部门内部不可控因素等原因而暂停。 但是前端团队内部做了多次培训，已经初步具备Electron及umi的能力

  5. 2020年3月左右，订货商城立项，经张波总（时任部门总负责人）同意，使用UMI技术栈来开发订货商城，并在项目中尝试我们关注的新特性。 在该项目中，我们熟悉了UMI的体系和原理，成功落地

  6. 2020年6月4号，在完成订货商城，图片优化，代码规范，前端文档开发任务后，升级所需的前置条件已经具备。 在和齐春波(时任部门技术总监)讨论后，从2020年6月5号起正式启动。

# 预期效果

  1. 通过工具加可控的手动修改代码，使程序稳定运行在新的框架上。
  2. 底层依赖升级到2020年6月前端行业的最新版本。
  3. 整个项目稳定运行在浏览器端。
  4. 整个项目稳定运行在桌面端（可选）。

# 技术能力

  1. 熟悉UMI的体系和原理
  2. 熟悉webpack, react, antd, babel，dva等依赖层各版本之间的不同
  3. 熟悉nodejs, 熟练使用node的API书写命令行工具
  4. 熟悉Electron（可选）
